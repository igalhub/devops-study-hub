## Overview
Firewalls control what traffic is allowed in and out of a system. Load balancers distribute that traffic across multiple backends. Together they're the entry point for every production system. Understanding both is essential for configuring access control, diagnosing connectivity failures, and building highly available architectures.

## Concepts

### Firewalls: The Core Idea
A firewall is a set of rules applied to network packets. Rules match on:
- **Source/destination IP** or CIDR range
- **Port** (destination or source)
- **Protocol** (TCP, UDP, ICMP)
- **Interface** (incoming or outgoing)
- **Connection state** (NEW, ESTABLISHED, RELATED, INVALID)

Rules are evaluated in order; the first match wins.

### iptables (Linux kernel firewall)
iptables is the traditional Linux firewall interface. On Ubuntu 20.04+ and RHEL 9+, the actual backend is `nftables` — `iptables` commands are translated via a compatibility shim (`iptables-nft`). The syntax and behavior below remain the same; the shim is transparent. It organizes rules into **chains** within **tables**:
- `filter` table (default) — accept/drop/reject packets
- `nat` table — address translation
- `mangle` table — packet modification

The three main chains in the `filter` table:
- `INPUT` — packets destined for this machine
- `OUTPUT` — packets generated by this machine
- `FORWARD` — packets passing through (routing)

```bash
# Show current rules (with line numbers and packet counts)
iptables -L -v -n --line-numbers
iptables -L INPUT -v -n

# Default policy (what happens if no rule matches)
iptables -P INPUT DROP    # deny by default (recommended)
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

# Allow established/related connections (essential — don't lock yourself out)
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

# Allow loopback
iptables -A INPUT -i lo -j ACCEPT

# Allow SSH
iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# Allow HTTP and HTTPS
iptables -A INPUT -p tcp -m multiport --dports 80,443 -j ACCEPT

# Allow from a specific IP
iptables -A INPUT -p tcp --dport 5432 -s 10.0.0.50 -j ACCEPT

# Drop everything else (explicit deny at end)
iptables -A INPUT -j DROP

# Save rules (persist across reboots)
iptables-save > /etc/iptables/rules.v4   # Debian/Ubuntu
service iptables save                     # RHEL/CentOS
```

### ufw — Uncomplicated Firewall (Ubuntu)
ufw wraps iptables with simpler syntax:
```bash
# Enable
ufw enable
ufw status verbose

# Allow / deny
ufw allow 22/tcp
ufw allow 80/tcp
ufw allow 443/tcp
ufw allow from 10.0.0.0/8 to any port 5432   # specific source
ufw deny 3306

# Delete a rule
ufw delete allow 80/tcp
ufw delete 3     # by rule number (from ufw status numbered)

# Allow by service name (reads /etc/services)
ufw allow OpenSSH
ufw allow 'Nginx Full'
```

### Cloud Security Groups (AWS/GCP/Azure)
Cloud firewalls are stateful, applied at the instance/NIC level:
- **Inbound rules:** what traffic can reach the instance
- **Outbound rules:** what traffic the instance can send (AWS defaults to allow all)
- **Stateful:** if you allow inbound on port 80, the reply is automatically allowed

AWS security group example:
```
Inbound:
  TCP  22    0.0.0.0/0        ← SSH (restrict this to your IP in production!)
  TCP  80    0.0.0.0/0        ← HTTP
  TCP  443   0.0.0.0/0        ← HTTPS
  TCP  5432  sg-0abc123def    ← PostgreSQL from app servers (reference another SG)

Outbound:
  All traffic  0.0.0.0/0      ← (default: allow all)
```

**Key difference from iptables:** security groups reference other security groups, not IPs — so "allow from the app server SG" works even as instances scale.

### Load Balancers: Types
| Type | Layer | Sees | Use case |
|---|---|---|---|
| L4 (Network) | TCP/UDP | IP + port | High throughput, any protocol |
| L7 (Application) | HTTP | URL, headers, body | HTTP routing, SSL termination, WAF |

**L7 load balancers** can:
- Route by URL path (`/api/*` → backend, `/static/*` → CDN)
- Route by hostname (`api.example.com` vs `app.example.com`)
- Terminate TLS (handle certs centrally, send HTTP to backends)
- Add headers (`X-Forwarded-For`, `X-Real-IP`)
- Health check backends and route around failures

### nginx as a Load Balancer
```nginx
upstream backend {
    # Round-robin (default)
    server 10.0.0.10:8080;
    server 10.0.0.11:8080;
    server 10.0.0.12:8080;

    # Weighted — 10.0.0.10 gets 3x the traffic
    # server 10.0.0.10:8080 weight=3;
    # server 10.0.0.11:8080 weight=1;

    # Least connections
    # least_conn;

    # Sticky sessions (same client → same backend)
    # ip_hash;

    # Health check (nginx Plus only; use with active checks)
    # keepalive 32;
}

server {
    listen 443 ssl;
    server_name api.example.com;

    ssl_certificate /etc/ssl/certs/api.example.com.crt;
    ssl_certificate_key /etc/ssl/private/api.example.com.key;

    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        proxy_connect_timeout 5s;
        proxy_read_timeout 30s;
    }
}
```

### AWS Load Balancers
| Type | Protocol | Use case |
|---|---|---|
| ALB (Application LB) | HTTP/HTTPS | Web apps, microservices, path routing |
| NLB (Network LB) | TCP/UDP/TLS | High-performance, static IPs, non-HTTP |
| CLB (Classic LB) | HTTP/TCP | Legacy, avoid for new deployments |

## Examples

### Diagnose a Blocked Port
```bash
#!/usr/bin/env bash
# Determine why a connection is being refused

TARGET_IP="$1"
TARGET_PORT="$2"

echo "1. Testing connectivity..."
nc -zv "$TARGET_IP" "$TARGET_PORT" 2>&1

echo "2. Checking local iptables..."
iptables -L INPUT -n | grep "$TARGET_PORT"

echo "3. Checking if service is listening locally..."
ss -tlnp | grep ":$TARGET_PORT"

echo "4. Tracing the route..."
traceroute "$TARGET_IP"
```
